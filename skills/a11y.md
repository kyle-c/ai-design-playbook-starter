# a11y.md

> Read before any interactive work. Accessibility is a build gate, not a polish phase.

## Standards

- **WCAG 2.1 AA minimum.** Aim for AAA on critical paths (auth, payments, error recovery).
- **Touch targets ≥ 44×44px** on mobile (Apple HIG / WCAG 2.5.5 enhanced).
- **Keyboard fully operable.** No mouse-only interactions, anywhere.
- **Screen-reader announcement** required for any state change the user can't see.

## Token-level audit (run before any screens use a token)

Every foreground/background pair in the **semantic** layer must be tested:

| Pair | Required ratio |
|---|---|
| Normal text (< 18px) | 4.5:1 (AA), 7:1 (AAA) |
| Large text (≥ 18px or 14px bold) | 3:1 (AA), 4.5:1 (AAA) |
| UI components & graphics | 3:1 |

If a pair fails:
1. Adjust hex at the **primitive** layer (one change fixes every component).
2. Suggest a hex that stays close to brand intent but passes.
3. Re-run audit. Don't move on with a known failure.

## Component-level audit

For every interactive element:

- [ ] Visible focus ring with `outline-offset: 2px` and ≥3:1 contrast against background.
- [ ] Reachable via Tab in logical reading order.
- [ ] Activatable with Enter (buttons, links) or Space (buttons, checkboxes).
- [ ] Has accessible name: visible label, `aria-label`, or `aria-labelledby`.
- [ ] Icon-only buttons MUST have `aria-label`.
- [ ] Loading state announced via `aria-busy="true"` or `role="status"`.
- [ ] Error state announced via `role="alert"` (interrupts) or `aria-live="polite"` (passive).
- [ ] Disabled state has both visual treatment AND `aria-disabled="true"` (prefer over `disabled` when an action explanation is needed).
- [ ] Touch target ≥ 44×44px including padding; adjacent targets have ≥ 8px gap.
- [ ] Animation respects `prefers-reduced-motion`.

## Forms

- Every input has a `<label>` (visible) or `aria-labelledby` (when label is visually a placeholder).
- Required fields marked with both visual indicator AND `aria-required="true"`.
- Error messages linked via `aria-describedby` AND announced via `role="alert"`.
- Error message structure: **what happened + why + what to do** (see `/skills/copy.md`).
- Don't validate on every keystroke; validate on blur or submit.
- Submit feedback: success state must be announced, not silent.

## Screen-level audit (run weekly + before any merge)

- [ ] Heading hierarchy is correct: one `<h1>`, no skipped levels.
- [ ] All images have meaningful `alt` text — or `alt=""` + `aria-hidden="true"` if decorative.
- [ ] Color is never the only signal (errors aren't only red; selected isn't only blue).
- [ ] Reading order on mobile matches visual order. Test with `Tab` and screen reader.
- [ ] Page is operable at 200% zoom without horizontal scroll.
- [ ] Live regions used sparingly; multiple competing announcements break SR users.
- [ ] No autoplaying media.
- [ ] Time limits, where present, can be extended or disabled.

## Motion

- Always provide a `prefers-reduced-motion` override (see `/skills/brand.md`).
- Replace transform/scale with opacity-only for users with reduced motion preferences.
- Avoid parallax, looping background animations, or anything with vestibular triggers.

## Tools

- Test contrast: [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/).
- Browser: axe DevTools or Lighthouse a11y panel — both free.
- Screen reader: VoiceOver (Mac, Cmd+F5) or NVDA (Windows, free).
- Keyboard test: literally just unplug your mouse for 10 minutes.

## Exceptions

The only acceptable a11y exceptions:
- Internal tools used by sighted, mouse-able operators only — and even then, document why in `/decisions/log.json`.
- Marketing motion that violates `prefers-reduced-motion` is not an exception. There is no exception.

## What good output looks like

- Tab through any screen with eyes closed; it should "feel" navigable.
- Voice over reads the page like a sentence: heading, then content, then actions.
- Error states tell you what to do next, in calm voice, never just red text.
- Focus rings are visible AND on-brand, not just blue browser default.
